Using Git with classes.csc.lsu.edu

The following notes describe the procedure for setting up your classes.csc.lsu.edu account to act as a Git server. All programming projects for this course will be stored on that server and all assignment submissions will be made using Git repositories there. This mechanism provides a central location to store all your code, while allowing easy access to that code for editing on your laptop, PC, or wherever.

For general guidance on the usage of Git, see the Git Book.

Set up the client

Download and install the Git client. In this class, we will focus on command-line Git usage because this is the most general and powerful Git interface. It is also the most consistent between Windows, OS X, and Linux, so everyone in the class will see the same behavior and your new skills will translate across platforms.

Once installed, a tiny bit of client configuration is required. Use the following commands to tell the Git client who you are. Your instructor is using his own name and email address as an example, and you must use your own name and email address to configure your own client.

git config --global user.name "Robert Kooima"
git config --global user.email kooima@csc.lsu.edu
git config --global push.default simple

You only need to do this once for your client. You would need to do this a second time only if you wish to set up a second machine as a Git client.

Set up a new repository

This course involves several programming projects. Each of these projects lives in its own Git repository. You create a new Git repository for each new assigment.

A repository is created on the Git server, in our case classes.csc.lsu.edu. To create a repository, you must SSH into your classes account. This example uses csc4356XX as login name.

slogin -l csc4356XX classes.csc.lsu.edu

Once there, create a directory for the repository. This example assumes you're doing the first programming project which will have the repository name "project1.git". You will eventually have "project2.git", "project3.git", etc.

mkdir project1.git

Move into this directory and initialize a "bare" git repository there.

cd project1.git
git --bare init

Now a new, empty Git repository exists in your account on the classes server.

Clone a repository

When you want to add content to a repository, you "clone" it using your local client, modify it by adding and changing files, and "push" the modifications to the server.

On your laptop or PC, move to a reasonable location for a working project and clone the project repository like so.

git clone cs4356XX@classes.csc.lsu.edu:project1.git

This creates a directory called "project1" in the current working directory.

Modify the working copy

Now you do whatever work that you need to do. Create files. Modify files. Delete or rename files. Do what needs to be done.

For this example, assume you've created a new file called README.md that describes the content of the project.

git status

Requesting the status of the project tells you that README.md is "untracked". That means Git doesn't care about it. First, let's add it to the set of files that Git cares about.

git add README.md

Now, the status will tell you that README.md has "Changes to be committed". Let's commit those changes to the repository. In so doing, we always include a message stating exactly what change we made.

git commit -m 'Added README'

With the commit complete, the local copy of your Git repository has embraced all of the additions you have made.

Push the changes to the server

Modifications to the project have been committed, but they have not yet been transmitted to the server. Synchronize the local copy with the remote server like so.

git push

With that, the copy of the project on classes.csc.lsu.edu includes all of your additions and changes. That remote copy is definitive, and it is accessible to the instructor and the grader. When the project deadline arrives, we'll clone it ourselves and evaluate the work on the basis of that clone.
